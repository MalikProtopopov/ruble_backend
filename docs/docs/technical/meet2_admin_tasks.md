Источник: `docs/mocks/2_meet.md`. Бэкенд: `meet2_backend_tasks.md`. Мобилка: `meet2_mobile_tasks.md`.

Документ описывает только админ-фронт.

## 1. Управление контентом сборов (приоритет)

Из ТЗ: «Подготовка к заполнению через админку контента». Это базовый блок — без него заказчик не сможет наполнять платформу.

### 1.1. Загрузка превью-картинок
- В форме создания/редактирования кампании добавить (или проверить наличие) загрузчика `thumbnail_url` — это и есть `preview_image_url`, который будет показывать мобилка.
- Эндпоинт уже есть (`PATCH /admin/campaigns/{id}` + media upload, см. недавний коммит c8c83d0 «Add image media type support»).

### 1.2. Загрузка стоковых видео
- В форме сбора — слот `video_url` (уже есть в модели Campaign).
- Поддержать загрузку файла + ввод URL.
- В отдельной вкладке/странице админки — справочник стоковых источников (просто markdown-страница со ссылками, наполняется руками заказчиком).

### 1.3. Документы сбора
- Уже в работе на текущей ветке (`backend/app/api/v1/admin/documents.py`, `005_documents.py`).
- В админке нужен раздел «Документы» внутри карточки кампании: загрузка картинок документов (отчёты, сертификаты), список с превью, удаление.
- См. также `documents_adminfront_guide.md` и `document_requirements_for_adminfront.md`.

## 2. Управление пользователями: новые состояния

После реализации анонимной регистрации в списке юзеров появятся новые статусы.

### 2.1. Фильтр и колонка «Тип аккаунта»
- В таблице пользователей добавить колонку `Тип`: `Гость` (`is_anonymous=true`), `Email` (`is_email_verified=true`), `Заблокирован` (`is_active=false`).
- Фильтр по типу.

### 2.2. Карточка юзера
- Показать `device_id` (если есть), `is_anonymous`, `is_email_verified`, `created_at`, число донатов, активна ли подписка.
- Отдельная секция «История объединений аккаунтов» — если этот юзер был результатом merge, показать timestamp merge и id source-аккаунта (если бэкенд хранит этот лог; см. раздел 5).

---

## 3. Управление подписками

### 3.1. Список подписок
- Существующий список (если есть) — добавить колонку `payment_method` (last4 + тип карты).
- Фильтр «активные / приостановленные / отменённые / pending payment method».

### 3.2. Действия над подпиской
- Просмотр истории `transactions` подписки (уже есть на бэке).
- Принудительная отмена.
- (На будущее) ручной запуск повторной попытки оплаты.

---

## 4. Управление сохранёнными методами оплаты

После реализации `payment_methods` (см. бэкенд раздел 5) — нужен админ-эндпоинт и UI:

- `GET /admin/users/{id}/payment-methods` — список карт юзера (только маскированные данные: last4, type, default, created_at).
- В карточке юзера — секция «Сохранённые карты» с возможностью **только просмотра и удаления** (не редактирования).
- Никаких полных данных карт. Только то, что хранит YooKassa.

## 5. Логи и аудит

### 5.1. Логи слияния аккаунтов
- Опционально: добавить таблицу `account_merge_log` (`source_id`, `target_id`, `merged_at`, `merged_by` — system).
- В админке — страница «История объединений», поиск по email/id.
- Это нужно для разбора инцидентов («куда делись мои донаты»).

### 5.2. Логи cooldown-блокировок
- Не обязательно. Если будет много жалоб «не даёт пожертвовать второй раз» — добавить простой лог 429-ответов в существующую систему логирования.

## 6. Контент стоковых видео

Из ТЗ: «Стоковые видео оставить — скинуть стоковые источники».

- Простая markdown-страница в админке (или отдельный раздел в админ-документации) со списком источников: Pexels, Pixabay, Coverr и т.д. Никакой логики — просто справочник.

---

## 7. Что НЕ нужно делать в админке (вынесено в мобилку)

Просто чтобы зафиксировать границы:
- Cooldown между донатами — настройка в `core/config.py`, **не** в админке (на старте). Если нужна гибкость — можно вынести в системные настройки позже.
- Логика merge аккаунтов — автоматическая, без ручного вмешательства админа.
- Push-уведомления юзерам — рассылаются автоматически, ручной рассылки через админку в этой итерации не делаем.

## Открытые вопросы

- Нужен ли админу инструмент ручного мерджа двух аккаунтов? (например, юзер написал в поддержку: «у меня два аккаунта»). На старте — нет, разруливаем через БД при необходимости. Позже — отдельная фича.
- Нужен ли просмотр истории cooldown-блокировок в админке? → по умолчанию нет, добавим если будет реальный спрос.
- Управление «3 сборами на сегодня»: курировать руками (явный флаг `is_featured` в кампании) или автоматически по `urgency_level`? → если курировать — нужен переключатель в карточке кампании.

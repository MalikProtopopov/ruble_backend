# По Рублю — Рекомендации по реализации платёжной логики
### Документ для согласования с заказчиком и передачи в разработку
**Версия 1.0 | Стек: FastAPI · PostgreSQL · Redis · Docker · ЮKassa**

---

## ЧАСТЬ 1. БИЗНЕС-ЛОГИКА
### Для согласования с функциональным заказчиком

---

### 1.1 Ключевое решение: подписка без баланса

**Рекомендация: реализовать рекуррентные подписки с прямым перечислением средств в НКО. Баланс пользователя внутри приложения — не делать.**

**Почему не баланс:**
По российскому законодательству (ФЗ-161 «О национальной платёжной системе», ст. 12) хранение предоплаченных денежных средств пользователей является деятельностью Оператора Электронных Денежных Средств. Эта лицензия выдаётся ЦБ РФ и требует капитала от 300 млн рублей. Ни одна благотворительная платформа в России не работает по такой модели — ни Tooba, ни Благо.ру, ни СберВместе, ни VK Добро. Нарушение этого требования — уголовная ответственность по ст. 172 УК РФ.

**Почему подписка работает:**
Рекуррентные автосписания регулируются иначе: пользователь один раз даёт согласие, и деньги автоматически списываются напрямую в НКО через платёжный агрегатор. Никакого промежуточного хранения на стороне «По Рублю» нет — значит, нет и регуляторных рисков.

**Что получает пользователь вместо баланса:**
Визуально в приложении отображается «Счётчик импакта» — виртуальный накопитель без реальных денег. Он показывает общую сумму всех пожертвований, текущую серию дней помощи (streak), конкретные достижения («Ваши 1 500₽ = 3 набора канцтоваров»). Это создаёт те же эмоции, что и баланс, но без юридических рисков.

---

### 1.2 Структура подписки

Пользователь оформляет одну подписку с параметрами:

**Сумма:** 1, 3, 5 или 10 рублей в день. Дефолт — 3 рубля.
По данным исследований, фрейминг «рублей в день» воспринимается в разы легче, чем эквивалентная годовая сумма. Четыре варианта — оптимальное количество (избегаем «паралич выбора»).

**Периодичность списания:** еженедельно или ежемесячно.
Технически важно: деньги не списываются каждый день. Если пользователь выбрал 3₽/день, с его карты раз в неделю снимается 21₽, раз в месяц — ~90₽. Причина — минимизация транзакционных комиссий и снижение вероятности failed payments. В интерфейсе это по-прежнему отображается как «3 рубля в день».

**Куда идут деньги:** пользователь выбирает одну из трёх стратегий при настройке подписки:

1. **«Платформа выбирает»** — деньги распределяются автоматически между наиболее приоритетными активными сборами. Рекомендуется как дефолт при регистрации.
2. **«Выбранный фонд»** — деньги всегда идут в конкретный верифицированный фонд (любой активный сбор этого фонда).
3. **«Конкретный сбор»** — привязка к определённому сбору (пользователь сам нашёл и открыл карточку).

Стратегия «Платформа выбирает» — дефолт. Исследования показывают: упрощение выбора кратно увеличивает конверсию. Пользователь может изменить стратегию в настройках в любой момент.

---

### 1.3 Что происходит при закрытии сбора

Это ключевой операционный сценарий, который **обязательно должен быть прописан в публичной оферте**.

**Сценарий A — подписка привязана к конкретному сбору:**
Система автоматически ищет следующий активный сбор того же фонда. Если находит — переключает туда и уведомляет пользователя push-сообщением: «Сбор для Арины закрыт — мы собрали нужную сумму! Теперь твой рубль помогает Ване». Если в фонде нет активных сборов — переходит к стратегии «Платформа выбирает».

**Сценарий B — нет ни одного активного сбора на платформе:**
Подписка временно приостанавливается. Деньги с карты не снимаются. Пользователь получает уведомление: «Все дети сейчас обеспечены — твой следующий рубль уйдёт, как только появится новый сбор». Streak не прерывается — ставится специальный флаг «приостановлено по причине отсутствия сборов».

**Почему не списывать в «резервный счёт»:**
Любое промежуточное хранение средств возвращает нас к юридическим рискам. Вариант Б — честный и понятный пользователю.

**Практическое решение:** на старте платформы держать минимум один «постоянный» сбор (например, «Помощь детям-сиротам» с обновляющимися историями), чтобы Сценарий Б не наступал. Это стандартная практика: charity:water никогда не закрывает The Spring, Teaming сохраняет «дефолтный» пул.

---

### 1.4 Уведомления пользователя — обязательный минимум

Все нижеперечисленные события требуют push/email уведомления. Это требование ФЗ-161 и базовый стандарт доверия:

- Каждое успешное списание (сумма, название сбора, дата)
- Неуспешное списание (с предложением обновить карту)
- Автоматическая смена сбора
- Приостановка подписки из-за отсутствия сборов
- Возобновление подписки при появлении нового сбора
- Закрытие сбора, в котором участвовал пользователь (радостное: «Собрали!»)

---

### 1.5 Что можно делать без юридических рисков

Следующие функции **безопасны** и не создают проблем:

- Несколько подписок одновременно (на разные фонды)
- Пауза подписки на период
- Отмена подписки в один клик (обязательно!)
- История всех пожертвований
- Визуализация суммарного вклада («счётчик импакта»)
- Streak — серия дней непрерывной помощи
- Шеринг достижений в соцсети

**Что не делать:**
- Любое хранение средств пользователя на стороне платформы
- «Кошелёк», куда можно зачислить деньги заранее
- Возможность «перевести остаток» между пользователями

---

### 1.6 Налоги и комиссии

Структура одного пожертвования (пример: 90₽/мес, тариф ЮKassa СБП):

| Статья | Сумма |
|---|---|
| Списано с карты пользователя | 90,00 ₽ |
| Комиссия ЮKassa (СБП 0,4%) | −0,36 ₽ |
| Комиссия «По Рублю» (15%) | −13,45 ₽ |
| Получает НКО | **76,19 ₽** |

Комиссия 15% — это выручка платформы, облагаемая как услуга (НДС 20% при ОСНО или без НДС при УСН). Пожертвования от НДС освобождены (пп. 12 п. 3 ст. 149 НК РФ).

Важно: пользователь может получить налоговый вычет 13% от суммы пожертвований (ст. 219 НК РФ), **только если средства перечислены напрямую в НКО**. Модель без баланса это обеспечивает автоматически.

## ЧАСТЬ 2. ТЕХНИЧЕСКИЕ РЕКОМЕНДАЦИИ
### Для разработчика бэкенда (FastAPI + PostgreSQL + Redis + Docker + ЮKassa)

---

### 2.1 Общая архитектура

Сервис состоит из трёх функциональных слоёв:

**API-слой (FastAPI)** — обработка запросов от мобильного приложения: создание/изменение/отмена подписок, получение истории, управление настройками аллокации.

**Payment-слой** — интеграция с ЮKassa: сохранение платёжных методов (токенизация), инициация списаний, обработка вебхуков о статусе платежей.

**Scheduler-слой** — фоновые задачи: периодический запуск биллинга, retry-логика, отправка уведомлений, синхронизация статусов сборов.

Разделение API-слоя и Scheduler-слоя в отдельные Docker-контейнеры — важно. Scheduler не должен зависеть от нагрузки на API и наоборот.

---

### 2.2 Схема данных

**Таблица `campaigns`** (сборы):
```
id, foundation_id, title, status (active/completed/paused/archived),
goal_amount, collected_amount, urgency_level (1–5), 
is_permanent (bool), ends_at, created_at, updated_at
```
Поле `is_permanent` — флаг для «вечных» сборов, которые никогда не закрываются автоматически. Используется для «резервного» сбора на старте.

**Таблица `foundations`** (фонды-партнёры):
```
id, name, legal_name, status (active/suspended),
yookassa_shop_id, created_at
```
У каждого НКО-партнёра должен быть свой `shop_id` в ЮKassa (или использоваться сплит-платежи через одного агрегатора с маршрутизацией).

**Таблица `subscriptions`** (подписки пользователей):
```
id, user_id, amount_kopecks (хранить в копейках!), 
billing_period (weekly/monthly), 
allocation_strategy (platform_pool/foundation_pool/specific_campaign),
campaign_id (nullable), foundation_id (nullable),
payment_method_id (токен ЮKassa),
status (active/paused/cancelled/pending_payment_method),
next_billing_at, created_at, cancelled_at
```
Сумму хранить в копейках (целое число) — никогда не хранить float для денег.

**Таблица `transactions`** (история списаний):
```
id, subscription_id, campaign_id (nullable), foundation_id (nullable),
amount_kopecks, provider_payment_id (id из ЮKassa),
status (pending/success/failed/skipped/refunded),
skipped_reason (nullable: no_active_campaigns/payment_method_invalid/...),
attempt_number, next_retry_at (nullable),
created_at, updated_at
```

**Таблица `allocation_changes`** (лог автоматических переключений):
```
id, subscription_id, from_campaign_id, to_campaign_id,
reason (campaign_completed/no_campaigns_in_foundation/manual),
notified_at, created_at
```
Этот лог нужен для двух вещей: поддержка пользователей («почему изменился фонд?») и аудит.

---

### 2.3 Интеграция с ЮKassa

**Схема первого платежа (токенизация):**

При первом пополнении пользователь совершает реальный платёж через SDK ЮKassa с параметром `save_payment_method: true`. ЮKassa возвращает `payment_method_id` (строка вида `pm-*`). Этот ID сохраняется в `subscriptions.payment_method_id`.

Важно: при первом платеже ЮKassa требует 3D Secure. Все последующие рекуррентные списания — без 3DS, в server-to-server режиме.

**Схема рекуррентного списания:**

Для каждого списания через API создаётся платёж с параметром `payment_method_id` и `capture: true`. Указываются `amount`, `description` и `metadata` (subscription_id, campaign_id, transaction_id — для трекинга). Платёж создаётся в статусе `pending`, ЮKassa асинхронно его обрабатывает и присылает вебхук.

**Вебхуки — критически важный компонент:**

Необходимо реализовать endpoint `/webhooks/yookassa` (POST). ЮKassa присылает уведомление при изменении статуса платежа: `payment.succeeded`, `payment.canceled`, `refund.succeeded`. Без обработки вебхуков статус транзакций не будет обновляться корректно. Endpoint должен: проверить подпись запроса (`X-Content-SHA256` header), найти транзакцию по `metadata.transaction_id`, обновить статус, запустить downstream-логику (уведомление пользователя, обновление счётчика, retry при отказе).

**Сплит-платежи (необходимо уточнить):**

Если каждое НКО имеет отдельный ЮKassa-аккаунт, можно настроить прямые переводы (пользователь → НКО, комиссия платформе). Это упрощает учёт. Если все средства идут на один счёт «По Рублю», нужна отдельная логика выплат НКО — технически сложнее и регуляторно чувствительнее.

**Рекомендация:** на старте — один общий ЮKassa-аккаунт «По Рублю» как посредника, еженедельные автовыплаты НКО. Это проще технически и понятнее юридически (платформа = агент НКО).

### 2.4 Resolver — функция выбора сбора

Это центральный алгоритм системы. Вызывается при каждом списании.

**Алгоритм для стратегии `platform_pool`:**
1. Получить все сборы со `status = active`, отсортированные по `urgency_level DESC`, затем по проценту заполнения `(collected_amount / goal_amount) DESC` (сначала почти закрытые — психологический эффект goal-gradient).
2. Исключить сборы фондов в статусе `suspended`.
3. Вернуть первый из списка. Если список пустой — вернуть `None`.

**Алгоритм для стратегии `foundation_pool`:**
1. Получить активные сборы конкретного фонда, та же сортировка.
2. Если нет — fallback на `platform_pool`.
3. Залогировать смену в `allocation_changes` и уведомить пользователя.

**Алгоритм для стратегии `specific_campaign`:**
1. Проверить статус привязанного сбора.
2. Если `active` — использовать его.
3. Если `completed/paused/archived` — искать следующий активный сбор того же фонда.
4. Если нет — fallback на `platform_pool`.
5. Логировать и уведомлять.

**Обработка `None` (нет активных сборов):**
Создать транзакцию со `status = skipped`, `skipped_reason = no_active_campaigns`. Деньги с карты не списывать. Сохранить дату следующей попытки. Пользователю — push с объяснением. Streak не прерывать.

### 2.5 Scheduler — биллинговый движок

Использовать **Celery + Redis** для фоновых задач (или ARQ как более легковесную альтернативу для FastAPI).

**Ключевые задачи:**

`task_process_billing` — главная задача биллинга. Запускается каждые 15 минут. Находит все подписки с `next_billing_at <= now()` и `status = active`. Для каждой вызывает resolver, создаёт транзакцию, инициирует платёж в ЮKassa. Обновляет `next_billing_at` на следующий период.

`task_retry_failed_payments` — retry-логика. Запускается каждый час. Находит транзакции с `status = failed`, `next_retry_at <= now()`, `attempt_number < max_attempts`. Реализует exponential backoff: первый retry через 24 часа, второй — через 3 дня, третий — через 7 дней, четвёртый и последний — через 14 дней. После 4 неудачных попыток — перевести подписку в `status = pending_payment_method` и уведомить пользователя.

`task_sync_campaigns` — при обновлении сборов через CMS/админку. Когда сбор переходит в `completed` — триггер на resolver для всех затронутых подписок.

`task_notify_streak` — ежедневный push пользователям с активным streak (можно выключать в настройках уведомлений).

---

### 2.6 Типы отказов платежей и логика обработки

Ответы ЮKassa содержат `cancellation_reason` — это ключевой параметр для retry-логики:

**Soft decline — повторять:**
- `insufficient_funds` — недостаточно средств → retry по расписанию
- `bank_declined_temporarily` — временный отказ банка → retry через 24 часа

**Hard decline — не повторять, уведомить пользователя:**
- `card_expired` — карта истекла → запросить обновление карты
- `payment_method_not_applicable` — метод не поддерживается
- `fraud_suspected` — подозрение на мошенничество

**Нейтральный — разобраться отдельно:**
- `three_d_secure_failed` — не должно возникать при рекуррентных (без 3DS), но если возникает — логировать отдельно

Логика важна: при `card_expired` не нужно делать 4 retry — это бесполезно и создаёт плохой UX (4 уведомления об отказе вместо одного понятного «обновите карту»).

### 2.7 Архитектура Docker

Минимальный набор контейнеров:

```
api          — FastAPI, обработка HTTP-запросов
worker       — Celery/ARQ worker, биллинг и фоновые задачи
beat         — Celery Beat, планировщик периодических задач
postgres     — основная база данных
redis        — брокер задач + кэш
```

Отдельный контейнер для `beat` — не совмещать с `worker`. Это позволяет масштабировать воркеры независимо.

**Переменные окружения обязательные для ЮKassa:**
```
YOOKASSA_SHOP_ID
YOOKASSA_SECRET_KEY
YOOKASSA_WEBHOOK_SECRET  # для верификации подписи вебхуков
```

**Важно по безопасности:** `payment_method_id` (токен карты) хранится в базе данных в открытом виде — это допустимо, сам по себе токен не даёт доступа к карте без секретного ключа ЮKassa. Секретный ключ ЮKassa — только в env, никогда в коде и не в Git.

---

### 2.8 Идемпотентность — обязательное требование

Все вызовы к ЮKassa должны быть идемпотентными. Для каждого платежа генерировать `idempotence_key` и сохранять его в `transactions` перед отправкой запроса в ЮKassa. Если запрос к ЮKassa завершился с сетевой ошибкой — при retry использовать тот же `idempotence_key`. ЮKassa вернёт результат предыдущего платежа, не создаст дублирующий.

Без идемпотентных ключей: сеть упала после отправки, но до получения ответа → при retry создаётся второе списание с пользователя. Это критический баг.

### 2.9 Что нужно в API (минимальный набор эндпоинтов)

**Подписки:**
- `POST /subscriptions` — создать подписку (инициирует первый платёж для токенизации)
- `GET /subscriptions` — список подписок пользователя
- `PATCH /subscriptions/{id}` — изменить настройки (стратегия, сумма)
- `POST /subscriptions/{id}/pause` — пауза
- `POST /subscriptions/{id}/resume` — возобновить
- `DELETE /subscriptions/{id}` — отменить

**Транзакции:**
- `GET /transactions` — история пожертвований (с фильтрацией)
- `GET /transactions/{id}` — детали платежа

**Счётчик импакта:**
- `GET /impact` — общая сумма, streak, достижения

**Вебхуки:**
- `POST /webhooks/yookassa` — приём уведомлений от ЮKassa

**Сборы (публичные):**
- `GET /campaigns` — список активных сборов
- `GET /campaigns/{id}` — детали сбора

### 2.10 Тонкости, которые стоит учесть заранее

**Часовые пояса:** хранить все даты в UTC в базе. `next_billing_at` — UTC. Для уведомлений конвертировать в timezone пользователя.

**Конкурентность биллинга:** при параллельных воркерах одна подписка может обрабатываться дважды. Использовать row-level locking (`SELECT FOR UPDATE SKIP LOCKED`) при выборке подписок для биллинга.

**Webhook delivery:** ЮKassa гарантирует доставку вебхука минимум один раз, но не ровно один раз. Обработчик вебхука должен быть идемпотентным — повторная обработка одного вебхука не должна создавать дублирующую транзакцию.

**Soft delete для подписок:** никогда не удалять записи физически. Использовать `status = cancelled` и `cancelled_at`. Для GDPR/ФЗ-152 реализовать отдельную процедуру анонимизации ПД через 3 года.

**Rate limiting:** ограничить создание подписок с одного user_id (например, максимум 5 активных одновременно). Ограничить попытки смены платёжного метода.

**Мониторинг:** ключевые метрики для алертов — процент failed payments за период, количество подписок в `pending_payment_method`, длина очереди задач в Redis, задержка выполнения биллинга (если `next_billing_at` просрочен более чем на 1 час для большого числа подписок).

### 2.11 Рекомендуемый порядок реализации (MVP → полная версия)

**MVP (launch):**
1. Таблицы `subscriptions`, `transactions`, `campaigns`, `foundations`
2. Первый платёж + токенизация через ЮKassa SDK
3. Ежемесячный биллинг (cron раз в месяц)
4. Resolver с единственной стратегией `platform_pool`
5. Обработка вебхуков ЮKassa
6. Базовые retry (2 попытки: через 24ч и через 7 дней)
7. Push-уведомления: списание успешно / неуспешно

**Post-MVP:**
8. Стратегии `foundation_pool` и `specific_campaign`
9. Полная retry-логика (4 попытки, разбивка по типу отказа)
10. `allocation_changes` лог + уведомления о смене сбора
11. Счётчик импакта и streak
12. Еженедельный биллинг как опция
13. Пауза/возобновление подписки
14. Несколько подписок одновременно

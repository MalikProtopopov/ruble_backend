## 📦 КОНТЕКСТ ПРОЕКТА (ground truth — не противоречь этому)

### Стек и слои

- **FastAPI** (async) + **SQLAlchemy 2.0 async** (`AsyncSession`) + **Pydantic v2**
  (`from_attributes=True`) + **Alembic** (миграции).
- **PostgreSQL** (asyncpg), **Redis** (rate-limit, троттлинг, кэш), **Taskiq + Redis**
  (фоновые задачи и крон-планировщик).
- **ЮKassa** — платёжный провайдер (разовые платежи, рекуррентные списания, вебхуки).
- **Firebase Admin** — push-уведомления. **S3/MinIO** — медиа (видео/аудио/фото/документы).
- **JWT RS256** (ключи в `backend/keys/`), `issuer="porubly"`, `audience="porubly-api"`.

**Поток обработки запроса (строго соблюдается):**

```
HTTP → Middleware (LastSeen) → Router (app/api/v1/...) → Depends (сессия БД, auth, пагинация)
     → Service (app/services/*.py — ВСЯ бизнес-логика) → Repository (app/repositories/*.py)
     → SQLAlchemy ORM (app/models/*.py) → PostgreSQL
Фон: Taskiq tasks (app/tasks/*.py), запускаются по крону или из сервисов/вебхуков.
```

Правила слоёв:
- **Роутер** — тонкий: только разбор запроса, auth-зависимость, логирование, вызов сервиса.
  Бизнес-логики в роутере нет.
- **Сервис** — вся логика, валидации бизнес-правил, обращения к репозиториям и внешним
  клиентам (ЮKassa, Firebase). Бросает доменные исключения.
- **Pydantic-схемы** (`app/schemas/<домен>.py`) — валидация входа и сериализация выхода.
- **Транзакция БД**: сессия живёт на запрос; внутри сервиса делается `session.flush()`
  (получить id, не коммитя), финальный `commit()` — на выходе из зависимости; при
  исключении — `rollback()`.

### Аутентификация и роли

- Токен передаётся в заголовке `Authorization: Bearer <access_token>` (RS256, TTL ~15 мин).
- **Анонимная авторизация по устройству**: `POST /auth/device-register` с `device_id`
  идемпотентно создаёт/возвращает пользователя (`is_anonymous=true`, refresh TTL 180 дней).
- **Привязка email через OTP**: `send-otp` → `verify-otp` (код argon2-хэш, TTL 10 мин,
  ≤5 попыток, rate-limit 5/60с через Redis). Привязка к анонимному аккаунту умеет
  **merge** существующего аккаунта (`allow_merge`).
- **Refresh-токены** хранятся как SHA-256-хэш в `refresh_tokens`, поддерживают `is_revoked`
  (logout).
- **Роли**: `donor` (обычный жертвователь), `patron` (генерирует платёжные ссылки),
  `admin` (админка). Зависимости (`app/core/security.py`):
  - `require_donor` — любой авторизованный пользователь;
  - `require_patron` — только роль `patron` (иначе 403);
  - `require_admin` — только роль `admin` (иначе 403);
  - `bearer_scheme` (опциональный) — когда auth не обязателен, но если токен есть — читаем `sub`.
- **Проверка владения** делается в сервисе через условие в запросе
  (`WHERE user_id == current_user AND is_deleted == false`), а не отдельным фреймворком.

### Формат ошибок (единый, из `app/core/exceptions.py`)

Все доменные ошибки — наследники `AppError`; хендлер отдаёт **единое тело**:

```json
{ "error": { "code": "DONATION_COOLDOWN", "message": "Текст для пользователя", "details": { } } }
```

Готовые классы и их HTTP-статусы:

| Класс / способ                         | HTTP | Когда                                              |
|----------------------------------------|------|----------------------------------------------------|
| `NotFoundError`                        | 404  | сущность не найдена / чужая                         |
| `ConflictError`                        | 409  | конфликт (email уже привязан и т.п.)               |
| `ForbiddenError`                       | 403  | нет прав на действие                               |
| `BusinessLogicError(code, message)`    | 422  | нарушено бизнес-правило (сбор не активен, мин. сумма)|
| `AppError(code, message, status_code)` | любой| произвольный (напр. `401 AUTH_REQUIRED`, `429 cooldown`) |
| `HTTPException` (FastAPI)              | 401/403 | отсутствует/невалиден токен, не та роль          |

`details` — машиночитаемый контекст для клиента (напр. `retry_after`,
`next_available_at`, `server_time_utc`). Сообщения для пользователя — **на русском**.

### Соглашения по API

- Префикс: `/api/v1`. Публичные ручки — `app/api/v1/public/*`, админские — `app/api/v1/admin/*`
  (все под `require_admin`), служебные — `auth.py`, `webhooks.py`, `media_proxy.py`,
  `payment_result.py`.
- **Коды статусов**: `200` (GET/PATCH), `201` (создание одной сущности), `204`
  (DELETE / logout без тела).
- **Пагинация — курсорная** (`app/core/pagination.py`): query-параметры `limit`, `cursor`;
  ответ — `{ "items": [...], "meta": { "next_cursor": "...", "has_more": true|false } }`
  через `paginated_response(...)`.
- **Именование схем** (`app/schemas/<домен>.py`): `XCreate` / `XUpdate` (вход),
  `XResponse` / `XDetailResponse` / `XListItem` / `XBrief` (выход). Все response-схемы
  наследуют `OrmBase` (`from_attributes=True`).
- **Идемпотентность платежей**: при создании доната/транзакции генерируется `idempotence_key`
  (uuid7), он же уходит в ЮKassa и хранится с `unique`-индексом → защита от дублей.
- **Деньги — всегда целое число копеек** (`amount_kopecks`). Минимальный донат — 100 коп.
  Суммы подписок ограничены набором (100/300/500/1000 коп.).
- **UUID v7** для первичных ключей. Таблицы с мягким удалением (`is_deleted`, `deleted_at`):
  users, donations, subscriptions, payment_methods.

### Доменные сущности (ключевые)

- **User** — донор/патрон; анонимность, email/верификация, стрик, суммарные донаты, `last_seen_at`.
- **Foundation** — фонд (ИНН, статус верификации, `yookassa_shop_id`).
- **Campaign** — сбор фонда; статус `draft|active|paused|completed|archived`, цель/собрано,
  кол-во доноров, `urgency_level (1..5)`, `is_permanent`, `ends_at`, «кулдаун» на повторный донат.
- **Donation** — разовый донат; суммы (платформенная комиссия / эквайринг / фонду), статус
  `pending|success|failed|refunded`, `source` `app|patron_link|offline`, `provider_payment_id`.
- **Subscription** — регулярное списание; `billing_period weekly|monthly`,
  `allocation_strategy platform_pool|foundation_pool|specific_campaign`, `next_billing_at`,
  статус `active|paused|cancelled|pending_payment_method`, `paused_reason`.
- **Transaction** — списание по подписке (с ретраями, `skipped_reason`, `attempt_number`).
- **PaymentMethod** — сохранённая карта (`provider_pm_id`, `card_last4`, `card_fingerprint`, `is_default`).
- Прочие: Achievement/UserAchievement, PatronPaymentLink, PayoutRecord, Document,
  ThanksContent, NotificationLog, MediaAsset.

### Фоновые задачи (Taskiq, крон)

Списание подписок (каждые 30 мин), ретраи неудачных транзакций (6 ч), авто-закрытие
истёкших сборов и пауза подписок (ежечасно), сверка платежей с ЮKassa и реконсиляция
«зависших» pending-донатов, push по стрикам/напоминания, чистки (OTP, refresh-токены,
логи), удаление неактивных анонимных аккаунтов. **В ТЗ обязательно указывай, какие
фоновые задачи затрагивает или порождает функция.**

## 📐 ФОРМАТ ТЗ (как именно писать каждый блок)

Для каждой функции выдавай ровно три блока. Заголовок функции — её бизнес-название.

### Блок 1. User Story

- **TL;DR для заказчика** (1–2 предложения простым языком, без терминов): что эта функция
  даёт пользователю. Это единственная часть, которую заказчик обязан понять с первого раза.
- **Роль и цель** по канону: *«Как `<роль: донор / патрон / админ>`, я хочу `<действие>`,
  чтобы `<ценность>»*.
- **Контекст/триггер**: где в приложении пользователь это делает, что предшествует.
- **Критерии приёмки** (Acceptance Criteria) — список в формате **Gherkin**
  (`Дано / Когда / Тогда`), покрывающий и happy-path, и ключевые ошибки.
- **Вне рамок (Out of scope)** — что эта функция НЕ делает (чтобы заказчик не ждал лишнего).

### Блок 2. API-запрос

Опиши контракт так, чтобы фронт мог интегрироваться без вопросов:

- **Метод и путь**: напр. `POST /api/v1/donations`.
- **Авторизация**: какая зависимость (`require_donor` / `require_patron` / `require_admin`
  / опционально `bearer_scheme` / без auth).
- **Заголовки**: `Authorization: Bearer <token>` (если нужен), `Content-Type`.
- **Path / Query параметры**: имя, тип, обязательность, дефолт, валидные значения.
- **Тело запроса (request body)** — таблица: поле, тип, обяз., ограничения, описание.
  Приведи **пример JSON**.
- **Ответы**:
  - **Успех** — код (`200/201/204`), схема ответа (имя `XResponse`), **пример JSON**.
  - **Ошибки** — таблица: HTTP-код, `error.code`, когда возникает, что в `details`.
- **Пагинация** (если список): параметры `limit`/`cursor` и форма `{items, meta}`.
- **Идемпотентность / побочные эффекты**, видимые клиенту (напр. возвращается `payment_url`,
  по которому надо открыть редирект ЮKassa).

### Блок 3. Бизнес-логика

Пошаговый алгоритм работы сервиса — нумерованным списком, в порядке выполнения:

1. **Входные проверки**: что валидируется на уровне схемы, что — в сервисе.
2. **Проверки прав/владения**: какое условие гарантирует, что пользователь имеет доступ.
3. **Бизнес-правила**: лимиты, кулдауны, статусы сущностей, допустимые суммы — с конкретными
   значениями и соответствующими `error.code`.
4. **Работа с БД**: какие таблицы читаются/пишутся, какие поля выставляются, статусы сущностей
   (`pending → success` и т.п.), индексы/уникальные ключи (идемпотентность).
5. **Внешние вызовы**: ЮKassa (создание платежа/рекуррент, `idempotence_key`, `return_url`,
   `metadata`), Firebase, S3 — что отправляем и что получаем.
6. **Побочные эффекты и фоновые задачи**: что произойдёт позже (вебхук подтвердит платёж →
   обновит статус → пересчитает аллокацию подписок; крон спишет подписку и т.п.).
7. **Транзакционность**: что должно быть атомарно, где `flush`, где `commit`, что откатывается.
8. **Краевые случаи**: дубликаты, гонки, недоступность провайдера, частичные сбои — и как
   система их обрабатывает (реконсиляция, ретраи).

В конце блока — **Влияние на существующий код** (какие сервисы/модели/миграции затрагиваются).

## 📎 ПРИЛОЖЕНИЕ. Эталонный пример ТЗ (по реальной ручке `POST /api/v1/donations`)

> Используй как образец качества и уровня детализации.

### Функция: Создание разового пожертвования

#### 1. User Story

**Как** авторизованный донор, **я хочу** пожертвовать выбранную сумму в конкретный сбор,
**чтобы** поддержать подопечного фонда и сразу перейти к оплате картой.

**Триггер:** экран сбора → кнопка «Помочь» → выбор суммы → подтверждение.

**Критерии приёмки:**
- *Дано* активный сбор и авторизованный пользователь, *когда* он отправляет донат на сумму
  ≥ 100 коп., *тогда* создаётся донат со статусом `pending` и возвращается `payment_url`
  для оплаты в ЮKassa.
- *Дано* сбор в статусе ≠ `active`, *когда* пользователь донатит, *тогда* `422 CAMPAIGN_NOT_ACTIVE`.
- *Дано* пользователь уже донатил в этот сбор недавно (кулдаун), *когда* он донатит снова,
  *тогда* `429 DONATION_COOLDOWN` с `details.next_available_at`.
- *Дано* запрос без токена, *когда* приходит донат, *тогда* `401 AUTH_REQUIRED`.

**Вне рамок:** подтверждение платежа (происходит асинхронно через вебхук ЮKassa);
регулярные списания (это подписки, отдельная функция).

#### 2. API-запрос

- **Метод/путь:** `POST /api/v1/donations`
- **Авторизация:** опциональный `bearer_scheme`, но при отсутствии `user_id` сервис бросает
  `401 AUTH_REQUIRED` → фактически нужен токен донора.
- **Заголовки:** `Authorization: Bearer <access_token>`, `Content-Type: application/json`.

**Тело запроса** (`CreateDonationRequest`):

| Поле                  | Тип       | Обяз. | Ограничения           | Описание                          |
|-----------------------|-----------|-------|-----------------------|-----------------------------------|
| `campaign_id`         | UUID      | да    | существующий сбор     | В какой сбор донат                |
| `amount_kopecks`      | int       | да    | ≥ 100                 | Сумма в копейках                  |
| `email`               | string    | нет   | email-формат          | Для чека (если не привязан)       |
| `payment_method_id`   | UUID      | нет   | своя сохранённая карта| Оплата сохранённой картой         |
| `save_payment_method` | bool      | нет   | дефолт `false`        | Сохранить карту после оплаты      |

```json
{ "campaign_id": "0192...", "amount_kopecks": 50000, "save_payment_method": true }
```

**Успех — `201 Created`** (`DonationResponse`):

```json
{
  "id": "0192...", "campaign_id": "0192...", "amount_kopecks": 50000,
  "status": "pending", "source": "app",
  "payment_url": "https://yoomoney.ru/checkout/payments/...", "created_at": "2026-05-31T13:30:00Z"
}
```

**Ошибки:**

| HTTP | `error.code`           | Когда                                  | `details`                         |
|------|------------------------|----------------------------------------|-----------------------------------|
| 404  | `NOT_FOUND`            | сбор не найден / карта не найдена       | —                                 |
| 422  | `CAMPAIGN_NOT_ACTIVE`  | сбор не в статусе `active`              | —                                 |
| 422  | `MIN_DONATION_AMOUNT`  | сумма < 100 коп.                        | —                                 |
| 429  | `DONATION_COOLDOWN`    | повторный донат в кулдаун-окне          | `next_available_at`, `retry_after`|
| 401  | `AUTH_REQUIRED`        | нет/невалиден токен                     | —                                 |

#### 3. Бизнес-логика

1. Если есть токен — `decode_token` → `user_id`. (Роутер только парсит и логирует запрос.)
2. **Сервис `donation.create_donation`**: загрузить `Campaign` по `campaign_id`;
   нет → `NotFoundError` (404).
3. Статус сбора ≠ `active` → `BusinessLogicError("CAMPAIGN_NOT_ACTIVE")` (422).
4. `user_id is None` → `AppError("AUTH_REQUIRED", status_code=401)`.
5. **Кулдаун**: `_check_donation_cooldown(user, campaign)`; если рано → `AppError(429, "DONATION_COOLDOWN")`
   c `details` (когда можно снова).
6. `amount_kopecks < 100` → `BusinessLogicError("MIN_DONATION_AMOUNT")`.
7. Если передан `payment_method_id` — проверить, что карта принадлежит пользователю и не удалена
   (`user_id == ... AND is_deleted == false`); иначе `NotFoundError`. Достать `provider_pm_id`.
8. `calculate_fees(amount)` → платформенная комиссия + сумма фонду. Сгенерировать `idempotence_key = uuid7()`.
9. Создать `Donation(status=pending, source=app, ...)`, `session.add` + `session.flush()` (получить id, без commit).
10. **ЮKassa** `create_payment(amount, description, idempotence_key, return_url=.../payment-result?donation_id=...,
    save_payment_method, payment_method_id, metadata={type:"donation", entity_id})`.
11. Записать `provider_payment_id`, `payment_url` в донат, `flush`. Вернуть объект → сериализация `DonationResponse` (201).
12. **Побочно (асинхронно):** ЮKassa шлёт вебхук `POST /api/v1/webhooks/yookassa` → `webhook_service`
    переводит донат в `success/failed`, наращивает `collected_amount`/`donors_count` сбора, может
    запустить пересчёт аллокации подписок. Зависшие `pending` подберёт крон-реконсиляция.

**Транзакционность:** запись доната и обновление его provider-полей — в одной сессии запроса
(`flush`, финальный `commit` на выходе); ошибка → `rollback`. Идемпотентность гарантируется
`unique`-индексом по `idempotence_key` (и в БД, и на стороне ЮKassa).

**Влияние на код:** `app/api/v1/public/donations.py`, `app/services/donation.py`,
`app/services/yookassa.py`, `app/services/payment.py` (комиссии), `app/services/webhook.py`,
модель `app/models/donation.py`.

### Как пользоваться этим промптом

1. Скопируй в LLM весь документ **до** раздела «Приложение» (системная инструкция +
   контекст + формат + шаблон + чек-лист). Эталонный пример можно тоже приложить — он
   повышает качество, но удлиняет промпт.
2. Допиши **шаблон ввода** (под-поля заполняют вход предсказуемо):

   ```
   ### Фича: <название>
   - Кто и зачем просил: <заказчик / боль пользователя>
   - Ожидаемое поведение: <2–4 строки, что должно происходить>
   - Затрагиваемые сущности: <Campaign / Donation / Subscription / ... — если знаешь>
   - Известные ограничения/правила: <лимиты, кто имеет доступ, что нельзя>
   - Приложенные файлы кода: <вставь app/services/*.py, app/schemas/*.py — или «нет»>
   ```

3. Модель выдаст: `🔍 Анализ → блок(и) по шаблону → ✅ чек-лист → ❓ вопросы`.
4. Ответь на `❓ Вопросы заказчику`, отдай ответы обратно в LLM — оно уберёт `⚠️` и
   зафиксирует финал. При сомнениях в именах полей — вставь нужный файл сервиса/схемы
   (модель сама попросит через `🔎 Пришли файл: ...`).

### Что усиливает результат (коротко)

- **Заземление:** факты — из контекста/кода; своё — помечать `💡 РЕШЕНИЕ`; неизвестное — `⚠️`.
- **Шаблон, а не проза:** модель копирует структуру надёжнее, чем следует описанию.
- **Сначала анализ, потом текст:** блок «🔍 Анализ» ловит ошибки до написания контракта.
- **Две аудитории:** `TL;DR` — для заказчика, таблицы/шаги — для разработчика.
- **Активная самопроверка:** чек-лист исполняется и даёт вердикт, а не лежит мёртвым грузом.

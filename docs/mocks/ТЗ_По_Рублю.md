## 1. Краткое резюме продукта

**Название:** По Рублю (Porubly)

**Назначение:** Благотворительная платформа микропожертвований, позволяющая пользователям жертвовать от 1 рубля на кампании некоммерческих организаций — разово или по подписке. Платформа агрегирует фонды и их кампании, автоматизирует распределение средств, обеспечивает прозрачность через отчётность и благодарственный контент от фондов.

**Целевая аудитория:**

| Роль | Описание |
|------|----------|
| **Донор (donor)** | Физическое лицо, совершающее разовые пожертвования или подписки. Взаимодействует через мобильное приложение |
| **Патрон (patron)** | Донор с расширенным статусом — может создавать платёжные ссылки для привлечения других доноров |
| **Админ (admin)** | Оператор платформы — управляет фондами, кампаниями, контентом, просматривает статистику и финансы |
| **Фонд (foundation)** | НКО, представленная на платформе. Не имеет собственного кабинета — управляется через админку |

**Границы системы (scope):**

| В scope | Вне scope |
|---------|-----------|
| REST API бэкенда (публичное + админское) | Мобильное приложение (Flutter — существует как UI-прототип с моками) |
| Интеграция с ЮKassa (платежи) | Личный кабинет фонда |
| Push-уведомления (Firebase FCM) | Веб-версия для доноров |
| Фоновые задачи (биллинг, реконсиляция, очистка) | Интеграция с бухгалтерскими системами |
| Хранение медиа (S3/MinIO) | Мультиязычность (i18n — описана в требованиях к документам, но не реализована) |
| Админское API для управления платформой | SMS-уведомления |
| Деплой через Docker Compose + Nginx + SSL | Аналитика и дашборды (кроме базовой статистики в API) |

## 2. Функциональные требования

### 2.1 Аутентификация и авторизация

- Система должна поддерживать аутентификацию пользователей (доноров/патронов) через одноразовый код (OTP), отправляемый на email.
- OTP-коды должны иметь ограниченное время жизни (10 минут) и rate-limiting (не более 5 запросов за 60 секунд).
- При первом успешном подтверждении OTP система должна автоматически создавать учётную запись пользователя.
- Система должна использовать JWT-токены с асимметричной подписью (RS256) — access-токен (15 мин) и refresh-токен (30 дней).
- Система должна поддерживать ротацию refresh-токенов с обнаружением replay-атак (повторное использование токена приводит к отзыву всех токенов пользователя).
- Система должна поддерживать отдельную аутентификацию администраторов по email + пароль.
- Система должна реализовать ролевую модель: donor, patron, admin — с разграничением доступа к эндпоинтам.

### 2.2 Управление пользователями

- Система должна позволять пользователю просматривать и редактировать профиль (имя, email, аватар).
- Система должна позволять пользователю настраивать параметры уведомлений (push_on_payment, push_daily_streak).
- Система должна позволять пользователю регистрировать push-токен (FCM/APNS) и часовой пояс.
- Система должна поддерживать анонимизацию аккаунта (удаление персональных данных в соответствии с ФЗ-152): email обезличивается, PII удаляются, подписки отменяются, токены отзываются.
- Админ должен иметь возможность просматривать, искать и управлять пользователями, менять роли (donor ↔ patron), деактивировать аккаунты.

### 2.3 Фонды (НКО)

- Система должна хранить информацию о благотворительных фондах: название, юридическое наименование, ИНН, описание, логотип, сайт, статус верификации.
- Каждый фонд должен иметь привязку к YooKassa (shop_id) для раздельного приёма платежей.
- Админ должен иметь возможность создавать, редактировать, верифицировать и приостанавливать фонды.
- Публичное API должно отдавать список активных (верифицированных) фондов.

### 2.4 Кампании

- Система должна поддерживать кампании сбора средств с жизненным циклом: черновик → активная → приостановлена → завершена → архив.
- Каждая кампания привязана к фонду и содержит: название, описание, видео, обложку, цель сбора, собранную сумму, количество доноров, уровень срочности (1–5), признак бессрочности, дату завершения.
- Система должна автоматически завершать кампании при достижении цели сбора.
- Система должна автоматически закрывать не-бессрочные кампании по истечении срока (фоновая задача).
- Админ должен иметь возможность досрочно закрыть кампанию с указанием причины.
- Публичное API должно предоставлять ленту активных кампаний с курсорной пагинацией, отсортированную по срочности и порядку сортировки.
- К кампании должны быть прикреплены: документы (PDF-отчёты) и благодарственный контент (видео/аудио).
- Админ должен иметь возможность регистрировать офлайн-платежи (наличные, банковский перевод) с защитой от дублирования.

### 2.5 Пожертвования (разовые)

- Система должна позволять авторизованным и гостевым пользователям совершать разовые пожертвования на кампанию.
- Минимальная сумма пожертвования — 1 рубль (100 копеек).
- При создании пожертвования система должна рассчитать комиссии (платформенная + эквайринг + сумма НКО) и создать платёж в ЮKassa.
- Гостевой flow должен требовать OTP-аутентификацию.
- Система должна предоставлять историю пожертвований пользователя с фильтрацией и пагинацией.

### 2.6 Подписки (рекуррентные платежи)

- Система должна позволять пользователю оформить подписку с фиксированной суммой (1, 3, 5 или 10 рублей).
- Максимум 5 активных подписок на пользователя.
- Биллинг-период: еженедельный или ежемесячный.
- Три стратегии распределения средств:
  - **specific_campaign** — на конкретную кампанию (с fallback на фонд при закрытии)
  - **foundation_pool** — на самую срочную кампанию фонда
  - **platform_pool** — на самую срочную кампанию на платформе
- При первом платеже система должна привязать карту через ЮKassa и сохранить payment_method_id для дальнейших автосписаний.
- Система должна поддерживать приостановку (вручную или автоматически при отсутствии кампаний), возобновление и отмену подписки.
- При закрытии целевой кампании подписка должна быть автоматически перераспределена на другую кампанию с логированием причины.

### 2.7 Биллинг и платёжная обработка

- Фоновая задача должна каждые 30 минут обнаруживать подписки, готовые к списанию, создавать транзакции и инициировать автоплатежи.
- Система должна обрабатывать вебхуки ЮKassa: `payment.succeeded` и `payment.canceled`.
- При успешном платеже: обновить статус, атомарно инкрементировать счётчики кампании (collected_amount, donors_count), обновить пользовательскую статистику (total_donated, streak), проверить благодарственный контент, начислить достижения, отправить push.
- При неудачном платеже: запланировать повтор с экспоненциальным отступом (1, 3, 7, 14 дней).
- Вебхуки должны проходить валидацию IP-адреса отправителя (официальные IP ЮKassa).
- Все платежи должны использовать ключи идемпотентности.

### 2.8 Благодарственный контент (Thanks)

- Фонды (через админку) могут прикреплять к кампаниям видео или аудио благодарности.
- Система должна показывать благодарности только тем пользователям, которые жертвовали на данную кампанию.
- Каждая благодарность отображается один раз (с возможностью повторного показа через 90 дней).
- При просмотре система рассчитывает персональный вклад пользователя (сумма, количество, даты).

### 2.9 Достижения

- Система должна автоматически начислять достижения по условиям:
  - streak_days (дни непрерывных пожертвований)
  - total_amount_kopecks (общая сумма)
  - donations_count (количество пожертвований)
- Проверка достижений происходит после каждого успешного платежа.
- Админ должен иметь возможность создавать и настраивать достижения (код, название, описание, иконка, условие, порог).

### 2.10 Патронские ссылки

- Патрон должен иметь возможность создать одноразовую платёжную ссылку для привлечения пожертвования на конкретную кампанию.
- Ссылка действует 24 часа, после чего автоматически истекает.
- При оплате по ссылке создаётся пожертвование, привязанное к патрону-создателю.

### 2.11 Медиа-файлы

- Система должна поддерживать загрузку файлов 4 типов: видео (до 500 МБ, mp4), документы (до 10 МБ, pdf), аудио (до 50 МБ, mp3/mp4/ogg/webm), изображения (до 20 МБ, jpeg/png/webp/gif/svg).
- Файлы сохраняются в S3-совместимое хранилище (MinIO) с уникальными ключами.
- Админ должен иметь возможность просматривать библиотеку медиа с фильтрацией по типу.

### 2.12 Документы (юридические/корпоративные)

- Система должна поддерживать CMS-подобное управление документами (политика конфиденциальности, оферта, отчёты).
- Документы имеют жизненный цикл: черновик → опубликован → архив.
- Каждый документ имеет: slug (URL-friendly), заголовок, аннотацию, HTML-контент, версию документа, дату документа, прикреплённый файл, порядок сортировки.
- Система должна поддерживать оптимистичную блокировку (поле version) для предотвращения конфликтов при одновременном редактировании.
- Публичное API предоставляет список опубликованных документов и доступ по slug.
- Мягкое удаление (soft delete).

### 2.13 Уведомления

- Система должна отправлять push-уведомления через Firebase Cloud Messaging (FCM) с поддержкой Android и iOS (APNS).
- 6 типов уведомлений: успех платежа (донору), успех транзакции (подписчику), завершение кампании, закрытие кампании, новая благодарность, ежедневное напоминание о стрике.
- Уведомления должны логироваться (получатель, тип, статус, ответ провайдера).
- Ежедневные стрик-уведомления отправляются в локальное время пользователя (12:00).
- При ошибке «невалидный токен» система должна автоматически очищать push_token пользователя.

### 2.14 Статистика и отчётность

- Админское API должно предоставлять сводную статистику платформы: GMV, комиссия, активные подписки, количество доноров, retention.
- Детальная статистика по кампании: пожертвования, транзакции, уникальные доноры, активные подписки.
- Система должна вести логи перераспределений подписок (allocation changes) и уведомлений.
- Ежедневные задачи реконсиляции должны проверять консистентность денормализованных счётчиков (collected_amount, donors_count, user impact) и логировать расхождения.

### 2.15 Выплаты фондам

- Админ должен иметь возможность фиксировать выплаты фондам (сумма, период, реквизиты перевода, примечание).
- Система должна рассчитывать задолженность перед фондом: сумма NCO из успешных пожертвований и транзакций минус уже выплаченное.

### 3.7 Что не реализовано

- **Тесты** — файлы тестов присутствуют (33 файла: 10 public API, 11 admin API, 7 integration, conftest), но степень покрытия не верифицирована.
- **i18n документов** — в требованиях описана таблица `document_locales`, но миграция и реализация отсутствуют.
- **Полнотекстовый поиск** — поиск по пользователям и фондам работает через `ILIKE`, без tsvector/GIN.
- **Rate limiting на все эндпоинты** — slowapi подключен, но применяется только к OTP.
- **Email-уведомления** — реализована только отправка OTP; нет email для платежей, подписок, кампаний.
- **Пагинация в отдельных админских списках** — часть списков отдаёт все записи без пагинации.
- **Аудит-лог действий админа** — нет общего журнала CRUD-операций админа.
- **Экспорт данных** — нет CSV/Excel-экспорта для отчётности.
- **API-версионирование** — только v1, но нет механизма для миграции на v2.
- **Webhooks для внешних интеграторов** — нет outbound-вебхуков для сторонних систем.

### 4.2 Безопасность

- **Аутентификация:** JWT RS256 с раздельными ключами (private/public). Access-токен 15 мин, refresh-токен 30 дней с ротацией и детекцией replay-атак.
- **Хеширование:** Argon2 для паролей админов и OTP-кодов.
- **Платёжные вебхуки:** Валидация IP отправителя по белому списку ЮKassa.
- **OTP rate limiting:** Не более 5 запросов за 60 секунд (Redis).
- **CORS:** Конфигурируемый список разрешённых origins.
- **Конкурентный доступ:** Pessimistic locking (`SELECT ... FOR UPDATE`) для лимитов подписок; оптимистичная блокировка (version) для документов.
- **Идемпотентность:** Все платежи используют idempotence_key (UUID).
- **PII:** Поддержка анонимизации данных (ФЗ-152).
- **Soft delete:** Пользователи и документы удаляются мягко, не теряя связей.

### 4.3 Логирование

- Структурированное логирование через structlog.
- JSON-формат в production, читаемый формат в development.
- Логгер настраивается per-module.
- Все бизнес-операции (платежи, подписки, аллокации, ошибки) логируются с контекстом.
- Задачи реконсиляции логируют расхождения с указанием entity_id и delta.

- [ ] Email-провайдер (SendGrid или SMTP) — отправка OTP-кодов
- [ ] PostgreSQL 16 — полная схема (22 таблицы, 20 ENUM)
- [ ] Redis — rate limiting, брокер фоновых задач

### 6.2 E2E-сценарии, которые должны проходить

**Аутентификация:**
- [ ] Запрос OTP → получение кода → верификация → получение JWT-пары → refresh → logout
- [ ] Replay-атака refresh-токеном → отзыв всех токенов пользователя
- [ ] Rate limiting OTP: 6-й запрос за 60 сек отклоняется

**Пожертвования:**
- [ ] Авторизованный донор → создание пожертвования → редирект на ЮKassa → вебхук succeeded → статус success, счётчики кампании обновлены, push отправлен
- [ ] Гостевой flow → OTP → пожертвование → вебхук → success
- [ ] Неудачный платёж → вебхук canceled → статус failed

**Подписки:**
- [ ] Создание подписки → привязка карты → первый платёж → вебхук → активация подписки
- [ ] Автобиллинг: cron создаёт транзакцию → рекуррентный платёж → вебхук → success
- [ ] Закрытие кампании → автоматическое перераспределение подписки → лог аллокации
- [ ] Пауза / возобновление / отмена подписки
- [ ] Лимит 5 подписок: 6-я отклоняется

**Кампании:**
- [ ] Лента активных кампаний с курсорной пагинацией
- [ ] Достижение цели сбора → автозавершение кампании
- [ ] Истечение срока → фоновая задача закрывает кампанию
- [ ] Полный жизненный цикл через админку: draft → publish → pause → complete → archive
- [ ] Досрочное закрытие → force-realloc → все подписки перераспределены

**Благодарности:**
- [ ] После пожертвования → unseen thanks доступны → просмотр → помечен как shown
- [ ] Повторный запрос unseen → пустой список

**Достижения:**
- [ ] Успешный платёж → проверка условий → начисление достижения

**Документы:**
- [ ] Создание черновика → редактирование → публикация → доступен по slug
- [ ] Оптимистичная блокировка: конфликт версий → ошибка 409

**Патронские ссылки:**
- [ ] Патрон создаёт ссылку → кто-то платит → donation привязан к ссылке → статус paid
- [ ] Ссылка не оплачена за 24 часа → cron-задача → статус expired

**Медиа:**
- [ ] Загрузка файлов всех 4 типов (видео, документ, аудио, изображение) → S3 → публичный URL
- [ ] Отклонение файла с превышением лимита или неверным типом

**Реконсиляция:**
- [ ] Искусственное расхождение collected_amount → задача реконсиляции логирует проблему

# Доработки мобильного приложения по итогам встречи 2

Источник: `docs/mocks/2_meet.md`. Бэкенд-часть: `meet2_backend_tasks.md`.

Документ описывает только мобильное приложение (Flutter). Для каждой задачи указаны новые/изменённые экраны и контракты API, которые мобилка должна потреблять.

---

## 1. Анонимная авторизация

### 1.1. Auto-register при первом запуске
- При первом запуске генерировать локальный `device_id` (UUID v4), **сохранять в платформенно-стабильном хранилище**:
  - **iOS**: Keychain с атрибутом `kSecAttrAccessibleAfterFirstUnlock` (плагин `flutter_secure_storage` с опцией `IOSOptions(accessibility: KeychainAccessibility.first_unlock)`).
  - **Android**: Keystore через тот же `flutter_secure_storage` (по умолчанию использует EncryptedSharedPreferences). Дополнительно — флаг `allowBackup=false` в манифесте для этого ключа, чтобы он не утекал в чужие резервные копии.
  - **Зачем именно так**: Keychain/Keystore переживают переустановку приложения, поэтому при reinstall мы попадём в **тот же** анонимный аккаунт (бэкенд по `device_id` отдаст того же юзера) и сохранённые подписки/карты не «осиротеют». Это закрывает основной кейс из бага «после переустановки моя подписка пропала».
  - **Не использовать** обычный `SharedPreferences` / документы приложения — эти хранилища стираются при удалении.
- При старте приложения, если нет access/refresh токена в storage → дёрнуть `POST /auth/device-register` с `device_id`, `push_token` (если уже выдан Firebase'ом), `push_platform`, `timezone`.
- Сохранить полученные access+refresh токены.
- Все последующие запросы идут с этим токеном — точно так же, как для зарегистрированного юзера.

### 1.2. Refresh без интервенции
- Refresh-токен у гостя живёт ~6 месяцев. На фронте логика обычная: 401 → дёрнуть `/auth/refresh` → повторить запрос.
- Если refresh умер → снова вызвать `device-register` с тем же `device_id`. Бэкенд вернёт того же юзера.

### 1.3. Экран «привязать почту» (опционально)
- Новый раздел в профиле «Сохранить аккаунт». Объясняет, что email нужен, чтобы не потерять историю при смене устройства.
- Поток: ввод email → `POST /auth/send-otp` → ввод кода → `POST /auth/link-email/verify-otp` (с текущим access токеном гостя).
- После успеха обновить локального юзера: `is_anonymous=false`, `email=...`. Показать тост «Аккаунт сохранён».
- Если бэкенд вернул, что email уже принадлежит другому юзеру → показать модалку «У вас уже есть аккаунт с этим email. Объединить?». При подтверждении → дёрнуть тот же endpoint с флагом `merge=true`. После успеха перелогинить пользователя в целевой аккаунт (новые токены придут в ответе).

### 1.4. Удалить блокеры в существующих экранах
- Убрать обязательность авторизации перед донатом. Сейчас (на встрече упоминалось) экран доната просит почту → надо снять это требование. Юзер уже залогинен через device-register, можно сразу платить.

## 2. Главный экран: список сборов

### 2.1. Новые поля в карточке сбора
Бэкенд начнёт возвращать в `GET /campaigns` (для авторизованных):
- `donated_today: bool`
- `last_donation: { id, amount_kopecks, created_at } | null`
- `next_available_at: ISO datetime | null`
- `has_any_donation: bool`

### 2.2. Визуал кнопки «Помочь»
- Если `donated_today == true` → кнопка в состоянии «Уже помог сегодня» (галочка, неактивный/disabled стиль).
- Если `next_available_at > now` → кнопка disabled с таймером «Можно через Xч Yм» (вычисляется на клиенте, обновляется раз в минуту).
- Иначе обычная активная «Поддержать».

### 2.3. Сортировка
Параметр `sort` в `GET /campaigns` (`default` / `helped_today` / `helped_ever`). Добавить переключатель/табы в UI. По умолчанию — `helped_today`.

### 2.4. Сброс «галочек» в полночь
Никакой клиентской логики не требуется — бэкенд считает `donated_today` по таймзоне юзера. Главное при pull-to-refresh снова дёрнуть список.

## 4. Экран сбора и cooldown

### 4.1. Обработка 429
При `POST /donations` бэкенд может вернуть `429` с телом `{ error: "donation_cooldown", retry_after, next_available_at }`.

- Показать дружелюбный bottom sheet «Спасибо! В этот сбор можно снова помочь через Xч Yм».
- Не дублировать как ошибку.

### 4.2. Превентивная блокировка
Не давать тапнуть кнопку «Помочь», если из списка уже известен `next_available_at > now` (см. 2.2). 429 — последняя линия защиты.

## 8. Saved payment methods

### 8.1. Новый раздел в профиле
- «Способы оплаты» → список карт из `GET /payment-methods`.
- Каждая карта: `**** 1234`, тип (Visa/MC/МИР), бейдж «Основная».
- Действия: «Сделать основной», «Удалить».

### 8.2. Сохранение при первом донате
- На экране ввода карты — чекбокс «Сохранить карту для будущих платежей» (по умолчанию включён).
- Передавать `save_payment_method: true` в `POST /donations`.

### 8.3. Быстрая оплата
- Если у юзера есть сохранённые карты — на экране доната по умолчанию выбрана дефолтная.
- При тапе «Поддержать» сразу `POST /donations` с `payment_method_id`, без редиректа в YooKassa.
- Опция «Оплатить другой картой» открывает обычный flow.

## 11. Защита от потерянных аккаунтов (orphaned anonymous)

> Добавлено после митинга 2026-04-08. Закрывает баг: после переустановки приложения у юзера пропадали подписка и сохранённая карта, потому что они оставались привязаны к старому анонимному аккаунту, к которому больше нет доступа.

Защита состоит из трёх линий обороны. **Главная** — стабильный `device_id` (см. 1.1). Два других — спасательные сценарии для случаев, когда `device_id` всё же сменился (например, юзер сменил телефон).

### 11.1. Стабильный `device_id` — главное условие
См. 1.1. Без этого пункта остальные не имеют смысла.

### 11.2. Recovery по карте (новый flow)

Когда мобилка привязала карту на свежем анонимном аккаунте (через донат с `save_payment_method=true` или через `bind-card`), бэкенд считает SHA-256 fingerprint от `first6+last4+expiry_month+expiry_year` и записывает его в `payment_methods.card_fingerprint`. Этот отпечаток одинаков для одной и той же физической карты, даже если YooKassa выдаёт разные `payment_method.id`.

**Эндпоинты — два варианта:**

#### Вариант A — scan по всем картам юзера 🟢 **рекомендуется для мобилки**

- `GET /api/v1/payment-methods/orphans` — берёт **все** non-deleted карты текущего юзера, по каждому fingerprint'у ищет анонимные аккаунты и возвращает объединённый список превью. **Не требует знать `pm_id`** — мобилке не нужно вылавливать ID свежесохранённой карты, не нужно ждать webhook YooKassa, достаточно дёрнуть после успеха любого платёжного flow.
- `POST /api/v1/payment-methods/recover` — слияние всех найденных по тем же fingerprint'ам. Идемпотентно.

Возвращаемые типы: `OrphanedAccountPreview[]` и `RecoveryResult` (см. ниже).

#### Вариант B — per-PM (точечный, оставлен для альтернативных flow)

- `GET /api/v1/payment-methods/{pm_id}/orphans`
- `POST /api/v1/payment-methods/{pm_id}/recover`

Семантика та же, но проверяет fingerprint только одной указанной карты. Используй, если по какой-то причине нужно явно ограничить scope (например, recovery в админке).

#### Схемы ответов

```json
// OrphanedAccountPreview[]
[
  {
    "user_id": "uuid",
    "donations_count": 12,
    "subscriptions_count": 1,
    "active_subscriptions_count": 1,
    "total_donated_kopecks": 36000,
    "last_seen_at": "2026-01-15T..."
  }
]
```

Учитываются только **анонимные, не удалённые** аккаунты. Если массив пустой — восстанавливать нечего.

```json
// RecoveryResult
{
  "merged_user_ids": ["uuid", ...],
  "donations_transferred": 12,
  "subscriptions_transferred": 1,
  "total_donated_kopecks_transferred": 36000
}
```

Источники soft-deleted, refresh-токены отозваны, всё переносится атомарно (как в `account_merge`). При повторном вызове `recover` массив `merged_user_ids` будет пустым — это нормально, эндпоинт идемпотентен.

**Мобильный flow (рекомендуемый — без знания pm_id):**
1. Юзер впервые на свежей установке делает донат с сохранением карты ИЛИ привязывает карту к подписке.
2. После того как мобилка получила пуш `donation_success` / `payment_success` (или poll'ом убедилась что donation/transaction перешёл в `success`) — дёрнуть `GET /api/v1/payment-methods/orphans`.
3. Если массив непустой — показать модалку:
   > **Найдены данные с прошлой установки**
   > Похоже, эта карта уже использовалась на этом устройстве. Мы нашли:
   > - 12 пожертвований на 360 ₽
   > - 1 активную подписку
   >
   > Восстановить их в текущем аккаунте? *Да / Нет*
4. Если «Да» → `POST /api/v1/payment-methods/recover`. После успеха обновить локального юзера, перезапросить `/me`, `/subscriptions`, `/payment-methods`, показать тост «Подписка восстановлена».
5. Если «Нет» → ничего не делать. Старые аккаунты будут вычищены кроном (см. 11.3).

**Когда дёргать `/orphans`:**
- ✅ Сразу после первого успешного донат-флоу с `save_payment_method=true`
- ✅ Сразу после успешного `bind-card` подписки
- ✅ Один раз при первом запуске после re-install (даже без новой карты — на случай если в Keychain что-то осталось)
- ❌ НЕ нужно дёргать на каждом запуске — fingerprint не появляется сам по себе, проверка имеет смысл только после изменения списка карт

**Важно:** Recovery работает только пока старый аккаунт ещё не вычищен (по умолчанию — 180 дней).

**Пограничные случаи (учтено на бэкенде, мобилке не нужно костылить):**
- Несколько orphan-аккаунтов с одной картой → все мерджатся за один вызов.
- Дублирующиеся активные подписки после merge → бэкенд автоматически отменяет лишние, оставляет самую старую (защита от двойного списания).
- Дублирующиеся `is_default` карты после merge → бэкенд оставляет default только на одной (самой свежей).
- Recovery от не-анонимного юзера (с email) → работает.
- PM без fingerprint (карта сохранена до миграции) → пропускается, не ломает flow.

### 11.3. Авто-уборка мёртвых аккаунтов (бэкенд) — что это значит для мобилки

Бэкенд раз в сутки запускает крон `cleanup_inactive_anonymous_users`:
- Условие: `is_anonymous=true`, `last_seen_at < now - 180 дней`.
- Действия: отменяет активные подписки, soft-delete'ит сохранённые карты, отзывает refresh-токены, шлёт push **«Подписка приостановлена. Войдите в приложение, чтобы восстановить»** (тип `subscription_expired_inactive`).
- Юзеры с историей донатов → soft-delete (история сохраняется). Юзеры без активности → hard-delete.

**Что должно делать мобильное приложение:**
- Обработать deep link из push типа `subscription_expired_inactive` — открывать экран «Подписки», показать баннер «Ваша подписка была приостановлена из-за длительной неактивности. Восстановить?». Кнопка ведёт на flow привязки карты → после успешной привязки автоматом дёрнуть `/orphans` (если recovery ещё возможно) или просто оформить новую подписку.
- На login-флоу проверять `users.is_anonymous` — если бэкенд вернул 401 (refresh умер) и `device-register` создал **нового** юзера (поле `is_new=true` в ответе), значит старый был вычищен. В этом случае имеет смысл сразу предложить юзеру привязать почту, чтобы такого больше не повторилось.

### 11.4. Tracking `last_seen_at`

Бэкенд автоматически (через middleware) обновляет `users.last_seen_at` на каждый аутентифицированный запрос (троттлинг — 15 минут на юзера через Redis). От мобилки **не требуется ничего дополнительного** — достаточно того, что приложение периодически делает запросы с access токеном (главный экран, профиль и т.д.).

> **Минимальная гигиена:** при открытии приложения не забывать дёргать любой авторизованный эндпоинт (например `GET /me`), чтобы продлить активность. Сейчас main flow это и так делает — отдельных правок не нужно.
